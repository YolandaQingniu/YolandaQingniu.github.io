Ordinary Bluetooth scale-self-managed Bluetooth

Ordinary Bluetooth scale-self-managed Bluetooth

This chapter is for users who have a certain understanding of Bluetooth and have their own Bluetooth framework in the app.

If you are not familiar with Bluetooth or do not have your own Bluetooth framework, this method is not recommended for access, otherwise it will increase the cost of access

One. Start Bluetooth scan

The customer starts and stops the Bluetooth scan by himself, and sends the scanned device information to our SDK (call QNBleApi.buildDevice). If it is our device, the SDK will Will build QNBleDevice, if it is not our device, it will return null/nil.

Usually the Bluetooth name of our device is QN-Scale or QN-Scale1, and we can usually filter our products by this Bluetooth name. After the target device is obtained, whether to stop scanning is determined by the APP. We generally recommend stopping Bluetooth scanning before making a connection.

Two. Create a Bluetooth protocol proxy class

Create your own Bluetooth protocol proxy class QNBleProtocolDelegate and implement all its methods. Specific implementation can be our Demo

three. Create Bluetooth protocol processing class

Create Protocol Processing Class QNBleProtocolHandler

four. Connect Bluetooth by yourself

Connected devices using system API or own Bluetooth framework.

The logic of this part is handled by the APP itself

Fives. Discovery Services、Enable eigenvalue

Our scales use two sets of services, one device will only contain one set. The APP needs to traverse all the services to find one set of service IDs, and if one is found, enable the feature values that require callback data according to the feature values in the table below.

After enabling the feature value, call [prepare]of QNBleProtocolHandler and pass the UUID of the service used

The first set

The service ID is: FFE0 Several eigenvalues are used below them, as follows:

UUID Attributes Description
FFE1 Notify Main data callback characteristic values
FFE3 Write Main data write characteristic value
FFE2 Indicate The secondary data callback feature value. Note that the attribute is Indicate. The Android enablement may be a little different.
FFE4 Write Secondary data callback feature
2A19 Read Charging scale battery characteristic value

The UUID of the characteristic value is only provided in the form of 32 bits. Android needs to use 128 bits.
For example, the 128 bits of FFE0 are: 0000ffe0-0000-1000-8000-00805f9b34fb

Second set

The service ID is: FFF0 There are only two characteristic values below it, one for data callback and one for writing data

UUID Attributes Description
FFF1 Notify Data callback characteristic value
FFF2 Write Data write characteristic value
2A19 Read Charging scale battery characteristic value

Data interaction

After enabling the characteristic value, you can receive the data uploaded by the scale.

After the APP receives the callback data from the scale, it sends it to the SDK for analysis through the method QNBleProtocolHandler.onGetBleData, and the parsed data will pass QNScaleDataListener callback.

In addition, if data needs to be written, the app will write on behalf of QNBleProtocolDelegate.writeCharacteristicValue.

Measurement completed

After the measurement is completed, the complete data will be called back through QNScaleDataListener, and then the scale end will actively disconnect the connection.

The next time you connect, you need to re-create QNBleProtocolDelegate and QNBleProtocolHandler
