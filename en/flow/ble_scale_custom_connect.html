Ordinary Bluetooth Scale - Manage Bluetooth Autonomously

Ordinary Bluetooth scale-Autonomous management of Bluetooth

This chapter is for users who have a certain understanding of Bluetooth and have their own Bluetooth framework in the app.

If you are not familiar with Bluetooth, or do not have your own Bluetooth framework, it is not recommended to use this method to access, otherwise it will increase the cost of access

First, start Bluetooth scanning

The customer initiates and stops the Bluetooth scan, and transmits the scanned device information to our SDK (call QNBleApi.buildDevice). If it belongs to our device, the SDK will Will build QNBleDevice, if it is not our device, it will return null/nil.

Usually the Bluetooth of our device is called QN-Scale or QN-Scale1, and we can usually filter our products by this Bluetooth name. After getting the target device, it is up to the APP to decide whether to stop scanning. We usually recommend stopping the Bluetooth scan before connecting.

II. Create a Bluetooth protocol proxy class

Create your own Bluetooth protocol proxy class QNBleProtocolDelegate and implement all its methods. The specific implementation can be our Demo

III. Create a Bluetooth protocol processing class

Create protocol processing class QNBleProtocolHandler using QNBleApi.buildProtocolHandler

IV. Connect yourself to Bluetooth

Connect devices using the system API or your own Bluetooth framework.

The logic of this part is handled by the APP itself.

五, Discover service, enable feature value

Our scale uses two sets of services, one device will only contain one set. The APP needs to traverse all the services to find one of the service IDs. If a set is found, the feature values ​​that need to be called back are enabled according to the feature values ​​in the table below.

After enabling the eigenvalue, call prepare of QNBleProtocolHandler and pass the UUID of the service used.

The first set

The service ID is: FFE0 Use some eigenvalues ​​below it, as follows:

UUID Attribute Description
FFE1 Notify Main data callback feature value
FFE3 Write The main data is written to the feature value
FFE2 Indicate Secondary data callback feature values, note that this property is Indicate, Android's enablement may be a bit different
FFE4 Write Secondary data callback eigenvalue
2A19 Read Charge scale power characteristic value

The UUID of the feature value is only provided in the form of 32 bits. If Android needs to use 128 bits, it will be stitched by itself. For example, the 128 bits of FFE0 are: 0000ffe0-0000-1000-8000-00805f9b34fb

The second set

The service ID is: FFF0 There are only two feature values ​​below it, one for data callbacks and one for data writes.

UUID Attribute Description
FFF1 Notify Data callback feature value
FFF2 Write Data write feature value
2A19 Read Charge scale power characteristic value

Data interaction

Once the eigenvalue is enabled, you can receive the data uploaded by the scale.

After the APP receives the callback data of the scale end, it sends it to the SDK for parsing by the method QNBleProtocolHandler.onGetBleData, and the parsed data will pass QNScaleDataListener callback.

In addition, if you need to write data, the APP will be written by QNBleProtocolDelegate.writeCharacteristicValue.

Measurement completed

After the measurement is completed, the complete data will be recalled by QNScaleDataListener, after which the scale will actively disconnect.

The next time you connect, you need to recreate QNBleProtocolDelegate and QNBleProtocolHandler
