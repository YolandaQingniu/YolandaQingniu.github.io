Ordinary Bluetooth scale-with Wifi function

Dual mode scale access guide

Introduction

The "dual-mode scale" mentioned here refers to a body fat scale with two data transmission functions: WiFi and Bluetooth

Function

Bluetooth

Set the ssid and password for WiFi

Data transfer for weighing

WiFi

Weighing data transmission

WiFi usage process

first time using

When using for the first time

1. Scanning device

Call the API of the ordinary Bluetooth scale, scan the Bluetooth device (QNBleDevice), and determine whether the value of isSupportWifi is true. If true, the Wifi function is supported.

2. Register Device

The WiFi device needs to register with the server and call the API register WiFi dual-mode device with Qingniu Cloud

This method requires a network, please ensure that the method is successfully executed, otherwise the cloud cannot receive the measurement data forwarded by our cloud The method can be successfully executed once, but it has no adverse effects multiple times.

3. Configure WiFi

The WiFi ssid and password are not saved at the scale end when leaving the factory. At this time, you need to call the relevant APP of Bluetooth to configure the scale end.

Networking API Please see Networking for WiFi Bluetooth Dual Mode Device

4. Writing network interfaces

Provide the address prefix of the client's cloud server, such as: http://www.baidu.com/scale/

We collectively refer to our company's servers as:Light Cow Cloud, the customer's server is: Merchant Cloud

Ask user profile interface

Scenes

Preconditions: The scale has been successfully registered, the scale has been configured with the network, and Bluetooth is not connected during the measurement.

At this time, the data will be uploaded to the light cow cloud via WiFi, and then the light cow cloud will forward the relevant information such as weight and measurement time to the merchant cloud.

At this time, the merchant cloud needs to query the user who is bound to the device in the service, and return the user information to the light cow cloud, and then the light cow combines the user data to calculate and then pass the data through Push Measurement Data Service.

Basic

Interface address: prefix + user_info, example: address is http://www.baidu.com/accept/user_info where user_info is appended to the provided interface address prefix

Request method: GET

Normally, the response will not be sent again after success, and it will not be sent again in the case of failure.

Push measurement data service

Scenes

After receiving the user data returned by Merchant Cloud, Qingniu Cloud will calculate the fitness data, and then push the data to Merchant Cloud through this interface

Basic

Interface address: prefix + measure, example: address is http://www.baidu.com/accept/measure where measure is appended to the provided interface address prefix

Request method: POST

During the notification interaction, if Light Cow received a response from the partner that was not successful or timed out, Light Cow thought that the notification failed, and Light Cow would periodically re-initiate the notification through a certain strategy to maximize the success rate of the notification, but Light Cow does not guarantee the notification. Finally succeed. (Notification frequency is 15/30/180/1800/3600, unit: second) In addition, in general, the response will not be sent again after a successful response. In the case of failure, the same notification may be sent to the partner system multiple times. Partner systems must be able to handle duplicate notifications correctly.

Signature Algorithm

The data interaction between Light Cow and the partner system will be digitally signed.At present, only the MD5 method is supported. When signing, you need to use the secret assigned when you enter Light Cow (a pair of partner_id/secret will be assigned when you enter Light Cow. secret is used for signature, please keep the key assigned by Qingniu properly) Parameters involved in signing: In the light cattle request parameter list, except for the sign parameter, the other parameters that need to be used are the parameters to be signed. The steps for signature generation are as follows:

  1. Put these parameters in a Map and remove the signature parameters:sign
  2. Sort the remaining parameters in ascending order of key (order from a to z, if the same first letter is encountered, see the second letter, and so on)
  3. After sorting is complete, press key&The value key-value pair format is concatenated into a string stringA, removing data where value is empty.
  4. Combine the above string with the secret allocated on the back to form a new string
  5. Use MD5 to sign the assembled string and verify that the signature is the sign in the parameter.

Pay special attention to the following important rules:

  1. If the parameter value is empty, do not participate in signature；
  2. When verifying the return of the call or proactively notifying the signature, the sign parameter transmitted does not participate in the signature, and the generated signature is checked against the sign value.
  3. Light cattle interface may add fields, and the extended fields must be supported when verifying signatures

During daily use

WIFI data interaction timing diagram

Normal measurement process

as follows:

  1. After the user measures on the scale, the data will be sent to the light cow cloud first
  2. According to Register Device Method, Light Cow Cloud judges that the device belongs to the customer, and then forwards the data to the cloud address that the customer has filled in.
  3. The customer cloud tries to determine the owner of the device according to the previous relationship between the device and the user. If it is determined, it returns the user's information to Qingniu Cloud.
  4. Qingniu Cloud received the user data returned by the customer cloud, combined with the data at the scale end, to calculate the complete measurement data.
  5. Light Cow Cloud pushes complete measurement data to the customer cloud.
  6. The customer cloud pushes the measurement data to the APP and displays the measurement data to the customer.

Matching to multi-user measurements Among the above steps, in step 3, it may find that the measurement user fails, and the following situations occur:

  1. No users are bound to this device In this case, the measurement data can be directly discarded to play the seven cow cloud, and the data is not matched.
  2. Multiple users are bound to the device at the same time, or multiple users are using the device at the same time. It is impossible to determine which user is using the scale, you can refer to the figure below, the user identification scheme adopted in our own APP.

  1. If a user can be uniquely identified through the scheme in the figure above, proceed according to the normal measurement process. If multiple users are still identified, refer to the following process.
