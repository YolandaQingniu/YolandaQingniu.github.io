FAQ

common problem

How to judge the type of equipment

Currently, the device type is only Normal Bluetooth Scale and Bluetooth Broadcast Scale. After calling Bluetooth scan, Normal Bluetooth device will return in onDeviceDiscover, Bluetooth broadcast scale will be called back in onBroadcastDeviceDiscover.

In addition, you can also easily judge by Bluetooth name. The body weight scale of Bluetooth named QN-Scale/QNScale1 is a common Bluetooth scale, and the Bluetooth name of QN-S3 is a Bluetooth broadcast scale.

appid and configuration file is what? What is the difference?

Appid is the unique identifier created by Light Cow for the customer. It will be registered in our background. A given appid will usually not change. The appid of 123456789 in the demo is the appid that we test and let the customer experience. The appid's features are unstable and may be modified at any time. If you have reached a formal cooperation intention, it is recommended that you apply for an appid for business or sales. The application process is very fast and can be completed in a few minutes.

The configuration file is some encrypted data used with the appid, and some information such as model, algorithm, and indicator are agreed upon. Different customer requirements will be different, so we package the customer customization requirements in this configuration file.

Can different clients use the same APPID?

Android and iOS can use an appid, applet/small plugin, and the appid of the SDK can't be used the same time.

Is the SDK offline? What data does it send to the Buffalo Cloud?

In the method initSdk, the SDK will send a request with our cloud check to see if the configuration file needs to be updated. If an update is required, the latest configuration file will be downloaded (the configuration file is small, typically 512 bytes), and if no updates are needed, no processing is done.

The SDK supports offline mode. If the customer rejects the request, we will set the configuration of the item in the configuration file to Offline. After being set to offline, the SDK will no longer send any network requests (this can be verified by the packet capture tool). The configuration file is no longer updated. If you need to add models or indicators, you need to apply for the configuration file again with our company, and then by the developer. Manual replacement.

Can I tell if the scale is off?

For 'normal Bluetooth scale' and 'dual mode scale', when the device is not connected, the device object QNBleDevice that is scanned is found to have an attribute isScreenOn, which indicates Whether the screen is bright. After connecting the device, it is not possible to directly judge whether the device is on the screen, but for some scales, the screen will be disconnected. The >isScreenOn attribute simply indicates that the device is bright at the moment the SDK is scanned. The status of the subsequent device screen changes, and QNBleDevice does not update dynamically. Need to be judged by scanning the new QNBleDevice callback.

For the 'broadcast scale', it is possible to scan the device, that is, it has been turned on, and it cannot be scanned, usually it is not turned on.

Data missing indicator returned after measurement is completed

If it is only the weight and BMI data can refer to no data such as body fat percentage after the measurement is completed.

If you can measure the body fat rate, but you are missing the indicator you want, the situation is usually not included in the configuration file, you need to communicate with our business / sales, increase the indicator.

After we add the indicator, we will resend a configuration file and need to replace the configuration file.

Send the model id of the scanned device to our company for confirmation.

No data such as body fat percentage after measurement

  1. Check whether the data is correct, mainly the two parameters of birthday (used to calculate age) and height, whether it is beyond the normal range.
  2. Is there any shoe removal for measurement? When measuring shoes, bioimpedance cannot be measured, so it is impossible to calculate indicators such as body fat percentage.

Can't scan the device, how to deal with it

  1. Check if Bluetooth is turned on, if there is a light scale, and if the phone is too far away from the scale (more than 10 meters)
  2. Check if the scanned device is connected by someone else.
  3. If it is Android, you can check if there is positioning permission, and if there is any open location service switch.
  4. Try restarting Bluetooth, no longer, try restarting Bluetooth.

Is there a bracelet SDK?

Our company has a smart bracelet device and also has an SDK. However, the SDK is not completely open to the outside world. If you are interested, you can contact our business negotiation.

Can I give a calculation method for indicators such as body fat percentage?

The algorithm for these indicators belongs to my driver and cannot be provided.

  • Check if the initialization file and the used appid match
  • Check if the imported SDK is up to date

The connection device has been unsuccessful or disconnected soon after success.

  • Check if the device is connected by someone else
  • Check if the currently connected device has been paired in the system Bluetooth, if you have already paired, you need to unpair
  • Some phones need to be scanned before they can connect successfully. Scan the device before connecting.

  • First determine if you have set up multiple monitors. Must be set to null when the listener is not in use
  • Determine if it is a shoe-wearing measurement, which may result in multiple measurements being completed in a short period of time

  • Check if both ACCESS_COARSE_LOCATION and ACCESS_FINE_LOCATION have been applied, and both permissions have been verified in the SDK.
  • Whether to compile version 26 and above, if yes, both permissions need to be applied separately (new features of 8.0)

    Android 6.0 or above, Google classifies Bluetooth as part of the positioning function. When using Bluetooth scanning, the user needs to authorize the positioning permission. Otherwise, the system will prompt the positioning permission when calling the scanning. A considerable number of mobile phones (about one-third of the way, when you do not open the location service switch, you can not scan the device, most of the original, and domestic systems will be optimized for this, not necessarily)

SDK returns the wrong file, confirms that there is no abnormality in the file location, and confirms that the file is used in the demo without exception.

  • Check if there is a so library added
  • Check if the packaged apk file contains a so library

What is the scale end storage data

When the ordinary Bluetooth scale is not connected to the mobile phone when weighing, the scale will store the data in the scale body. When the Bluetooth is connected next time, the data will be transmitted to the APP. Since the SDK cannot identify who the user is, it will put This data is assigned by the APP, and the assigned user can hand it over to the SDK to calculate the complete data.

Can I provide Bluetooth protocol?

In principle, we will not provide a Bluetooth protocol. If it is really needed, we can communicate with our business/sales.

Why do we use the data measured by the SDK to differ from the data of the light or runcobo health?

We have several algorithms, different algorithms that different devices may use.

In addition, we have a health problem mechanism in our own app that will allow users to answer 2 questions and use different algorithms depending on the user's response.

The algorithmic mechanism in the SDK is usually a unified algorithm that is immutable. That is to say, the same scale uses our APP measurement and uses the SDK to measure. The calculation data is the same as the weight, and there may be many differences. Therefore, our company does not recommend using the measurement data of our APP and SDK for comparison. Customers only need to care whether the given algorithm is suitable and stable.

After the Bluetooth version is abnormal in the online version, how to troubleshoot the problem

First check the user's mobile phone model, if possible, try to use the same model as the customer to test.

Register the log output interface QNLogListener, record the relevant logs (can save the log file), and send it to our developers for analysis.

SDK can only do protocol parsing, Bluetooth scanning and connection are managed by the customer

This is achievable, and the implementation of the normal Bluetooth scale and the Bluetooth broadcast scale will be different.

Self-implementation of Bluetooth scanning and connection management, there are certain technical difficulties, if there is no relevant Bluetooth development experience, it is not recommended to do this.

Normal Bluetooth Scale

  1. Write your own Bluetooth Protocol Proxy Class to implement various Bluetooth operation methods.
  2. Write the Data Listening Callback Class QNScaleDataListener and register it to the SDK using [QNBleApi.setBleDeviceDiscoveryListener.
  3. Perform a Bluetooth scan yourself and pass the scanned information to the SDK. Use the method QNBleApi.buildDevice to create a light blue Bluetooth device object.
  4. Create a user object using the method QNBleApi.buildUser
  5. Create Bluetooth Protocol Processing Class
  6. Make a Bluetooth connection
  7. After the connection is successful, after discovering that the service is successful, call the default method of the Bluetooth protocol processing class
  8. Process the data of QNScaleDataListener callback
  9. Collect the stable data and handle it yourself.

Bluetooth Broadcasting Scale

When will the Bluetooth of the body fat scale be turned on, and how long will it automatically turn off the Bluetooth (or disconnect the phone)?

Normally, the body fat scale Bluetooth will turn on when the scale is on (that is, when the screen is lit), and the broadcast will be broadcast. After the measurement is finished, the scale will meet the screen, and the screen will generally continue to connect for 1-2 minutes (depending on the production batch), and then actively disconnect the measurement. This mechanism is for the user to lift the scale again in a short time without having to connect again. If the APP needs to be disconnected immediately after measurement, it can be actively disconnected after a delay of more than 1 s after receiving stable data.

The status of the Scale Status Definition of your official website API is 0-9. Why is there a status of "-1" in the Demo you provided (and it really returns to this state)

-1 always exists, indicating that the default state is lost, that is, the connection is disconnected. Usually received after the scale is actively disconnected. We will unify this state to the disconnected state as soon as possible.

After searching for Bluetooth and connecting it, the foot stays on the body fat scale for a few seconds, then removes the foot and returns the status code of the measurement completion (there is no data returned in the method of returning data)

Why can we search for Bluetooth without any operation (no one is close to the body fat scale within 10-30 minutes)

When weighing, if the APP is connected, the scale will cache this data into it, and the broadcast will also be broadcast for 5-30 minutes (depending on the production batch). After the APP is connected, the cached data can be received and displayed to the client. This function is used to solve the problem. When the user weighs, the mobile phone is not used. After the weighing is finished, sit on the sofa and open the mobile phone and receive the data.
